## Задание 1. Учебный проект Mesto

Проект представляет собой простую реализацию приложения для загрузки и просмотра фотографий.
Пользователи могут регистрировать аккаунты, загружать и удалять свои фотографии и отмечать понравившиеся фотографии, загруженные другими пользователями.

### Ограничения

При анализе решения были выявлены следующие особенности, подлежащие доработке в следующих версиях:
1. Предоставленный фронтенд работает с удаленными API:
* https://nomoreparties.co/
* https://auth.nomoreparties.co/
2. https://nomoreparties.co/ не реализует полноценную аутентификацию, т.к. работает с подставляемой из кода заглушкой токена. Наличие jwt-токена, полученного из auth.nomoreparties.co проверяется только на фронте.
3. предоставленный фронтед не может без доработки приложения в целом быть подключен к предоставленному бэкенду, развернутому локально, т.к.: 
* локальный бэкенд требует авторизации с помощью jwt;
* ожидает куки, а фронтенд хранит токен в LocalStorage;
* предоставляет пути к API, не включающие в себя GroupId;
* не обрабатывает cors.
4. Существует проблема с общим языком в терминах DDD: фотографии в постановке задачи называются фотографиями(фото), а в коде - карточками, картинками и местами. В настоящем описаниии я буду использовать слово "фотография"
5. Указанные выше проблемы не будут исправлены в ходе работы над версией, предполагается запускать только фронтенд-приложение в связке с сервисом nomoreparties.

### Описание текущей реализации
Текущая реализация приложения Mesto может быть описана следующей диаграммой:
Диаграмма приведена на первой вкладке файла [по ссылке](https://drive.google.com/file/d/1fUBf5ZMIQtVUciJ0VMNe06cKPMqwDyR-/view?usp=drive_link)

Монолитный фронтенд обращается, судя по всему, к двум сервисам:
1. Сервису аутентификации и регистрации (auth.nomoreparties.co)
2. Сервису основной логики приложения (nomoreparties.co).

Мне неизвестно, как именно общаются эти сервисы между собой для синхронизации данных о профилях, в рамках этой работы предположим, что они разделяют общую БД.

### Описание изменений
Для реализации микрофронтендов выбран подход Module Federation, т.к. Mesto - на данном этапе небольшое приложение, не использующее различных фреймворков, т.е. подход Single SPA кажется избыточным. Кроме того, если обнаружатся общие зависимости у микросервисов, будет удобно сделать их разделяемыми.

Исходя из бизнеса, можно выделить следующие домены:

* регистрация и авторизация пользователя
* работа с профилем пользователя;
* работа с фотографиями.

И их сущности:
* профиль пользователя;
* фотография.

Т.о. предполагается создать следующие микрофронтенды.

1. auth
* Регистрация пользователя
* Фвторизация пользователя

2. users:
* получение данных профиля;
* редактирование профиля;
* редактирование аватара.

3. cards:
* получение и отображение списка фотографий;
* добавление фотографии;
* удаление фотографии;
* добавление лайка;
* удаление лайка;
* учет лайков.

4. ui_components
* Предоставление разделяемых компонентов ui

Кроме того выделен в отдельную библиотеку расшаренный контекст user_context

Реализация отображена на схеме [по ссылке](https://drive.google.com/file/d/1fUBf5ZMIQtVUciJ0VMNe06cKPMqwDyR-/view?usp=drive_link). Схема расположена на второй вкладке.

### Установка
Установка и развертывание будут описаны только для фронтенд приложения.

Шаги для установки в докере:
1. Склонируйте репозиторий
```bash
git clone git@github.com:PrevedMed6/architecture-sprint-1.git
```
2. Перейдите в папку репозитория
```bash
cd .\architecture-sprint-1\frontend\microfrontend\
```
3. Запустите docker-compose
```bash
docker-compose up --build
```
